Master Setup Guide
==================

Last updated: FSM definition-to-runtime interpreter flow, transition auto-generation outputs, and per-issue validation logging documented

FSM data flow (authoritative): definition asset -> runtime interpreter
- Authoring source of truth is `StateMachineDefinition` assets (`initialStateId`, `states`, `transitions`).
- At runtime, `UnitBrain` bootstraps `StateMachineRuntime` and resolves one machine by:
  1) directly assigned `MachineDefinition`, or
  2) `MachineDefinitionId` lookup through `StateMachineRegistry`.
- `StateMachineRuntime` acts as the interpreter for definition data:
  - builds state lookup tables from `StateMachineDefinition.States` and `UnitBrain.LegacyStateMappings`,
  - builds transition lookup tables from `StateMachineDefinition.Transitions`,
  - resolves transition eligibility from `eventName`, optional `conditionId` (`ConditionRegistry`), and legacy `conditionDescription` tokens.
- Runtime start state selection order is:
  1) `StateMachineDefinition.InitialStateId`,
  2) first declared definition state,
  3) `UnitBrain.LegacyInitialState` (migration fallback),
  4) deprecated `UnitBrain.InitialState` (compatibility fallback only).

FSM transition auto-generation (editor automation):
- Entry point: `Tools/FSM/Generate Transition Assets from Definitions`.
- Generator script: `Assets/Scripts/Editor/FSM/StateMachineTransitionGenerator.cs`.
- Read model: `StateMachineDefinition.transitions` (`fromStateId`, `toStateId`, `eventName`, `conditionId`, `conditionDescription`).
- Output location: generated `BehaviourTransition` assets are written to:
  - `Assets/FSM/Transitions/<machineId>/<from>_to_<to>__<stableHash>.asset`
- Stable key includes machine + from + to + event/condition/payload tokens, so reruns are deterministic/idempotent.
- On each run/import cycle, generator behavior is:
  - update existing generated assets,
  - create missing generated assets,
  - delete stale generated assets no longer represented by definitions.
- Generated assets are marked with:
  - label `FSM-AutoGeneratedTransition`,
  - generated metadata fields on `BehaviourTransition` (`isAutoGenerated`, machine/from/to IDs, condition/payload, stable key, generated comment).
- Synchronization is also triggered automatically on domain reload and when `StateMachineDefinition` assets are imported/moved/deleted.

FSM hierarchical transition lifecycle semantics:
- `UnitBrain` computes old/new ancestry chains and performs LCA-based traversal.
- Parent ancestry source order:
  1) `StateDefinition.parentStateId` via runtime definition mapping,
  2) fallback to legacy `BehaviourState.Parent` when no definition parent mapping exists.
- During transition:
  - `OnExit` executes from old leaf upward (excluding LCA),
  - `OnEnter` executes from below LCA down to the new leaf.

Migration and deprecation notes:
- `UnitBrain.LegacyStateMappings` remains migration glue for mapping `StateDefinition` IDs to legacy `BehaviourState` assets.
- `UnitBrain.LegacyInitialState` remains a temporary migration fallback.
- Deprecated direct wiring fields (`InitialState`, `Transitions`) are compatibility-only and must not be used for new content authoring.
- `BehaviourTransition` assets are not the authoritative runtime path; generated copies exist for migration visibility/tooling workflows.

Validation architecture notes:
- Unity menu: `Tools/Validation/Validate All Definitions`.
- Runtime bootstrap, editor menu validation, and CI use the same validation orchestration core.
- CI execute method: `DefinitionValidationMenu.ValidateAllForCI`.

Validation logging semantics (authoritative):
- `DefinitionValidationOrchestrator.RunValidationAndLog()` now emits one Unity console log entry per validation issue.
- Severity routing is explicit:
  - `ValidationIssueSeverity.Error` -> `Debug.LogError`
  - `ValidationIssueSeverity.Warning` -> `Debug.LogWarning`
  - `ValidationIssueSeverity.Info` -> `Debug.Log`
- Each issue log line includes: validation code, registry, message, and optional asset metadata (`AssetPath`, `AssetId`, `Field`) when available.
- The run ends with a compact summary line in the format: `[Validation] N issue(s) / M error(s).`
- Editor and CI entry points continue using `report.HasErrors` / `report.ErrorCount` for pass/fail behavior.
