Master Setup Guide
==================

Last updated: UnitBrain transitions now perform parent-aware hierarchical OnExit/OnEnter lifecycle calls

FSM data flow (authoritative): definition asset -> runtime interpreter
- Authoring source of truth is `StateMachineDefinition` assets (`initialStateId`, `states`, `transitions`).
- At runtime, `UnitBrain` bootstraps `StateMachineRuntime` and resolves one machine by:
  1) directly assigned `MachineDefinition`, or
  2) `MachineDefinitionId` lookup through `StateMachineRegistry`.
- `StateMachineRuntime` acts as the interpreter for definition data:
  - builds state lookup tables from `StateMachineDefinition.States` and `UnitBrain.LegacyStateMappings`,
  - builds transition lookup tables from `StateMachineDefinition.Transitions`,
  - resolves transition eligibility from `eventName`, optional `conditionId` (`ConditionRegistry`), and legacy `conditionDescription` tokens.
- Runtime start state selection order is:
  1) `StateMachineDefinition.InitialStateId`,
  2) first declared definition state,
  3) `UnitBrain.LegacyInitialState` (migration fallback),
  4) deprecated `UnitBrain.InitialState` (compatibility fallback only).

FSM hierarchical transition lifecycle semantics:
- `UnitBrain` now computes full ancestry chains (`state`, `Parent`, ... root) for both current and target states.
- During transition, it finds the lowest common ancestor (LCA) between old and new chains.
- `OnExit` is called from the old leaf upward, excluding the LCA.
- `current` is then set to the transition target state.
- `OnEnter` is called from just below the LCA downward to the new leaf.
- This guarantees parent states (for example `WorkingState`) participate consistently when entering/exiting child states.

- `UnitBrain.EmitEvent(evt)` first offers events to current state and parent chain via `HandleEvent`.
- A state only blocks transition evaluation when it returns `true` from `HandleEvent`.
- Transition-driving events in worker leaf states are intentionally not consumed internally.
- Worker leaf states (`MoveToResourceState`, `GatheringState`, `MoveToDropoffState`, `DeliveringState`) now return `false` for `Arrived`, `WorkComplete`, and `DeliveryComplete` so transition assets can execute.

Verified worker transition chain:
- `Idle -> MoveToResource` on `OnOrderGather`
- `MoveToResource -> Gathering` on `Arrived`
- `Gathering -> MoveToDropoff` on `WorkComplete`
- `MoveToDropoff -> Delivering` on `Arrived`
- `Delivering -> Idle` on `DeliveryComplete`

Legacy validation architecture notes:
- Unity menu: `Tools/Validation/Validate All Definitions`
- Runtime bootstrap, editor menu validation, and CI use the same validation orchestration core.
- CI execute method: `DefinitionValidationMenu.ValidateAllForCI`.

Validation logging semantics (authoritative):
- `DefinitionValidationOrchestrator.RunValidationAndLog()` now emits one Unity console log entry per validation issue.
- Severity routing is explicit:
  - `ValidationIssueSeverity.Error` -> `Debug.LogError`
  - `ValidationIssueSeverity.Warning` -> `Debug.LogWarning`
  - `ValidationIssueSeverity.Info` -> `Debug.Log`
- Each issue log line includes: validation code, registry, message, and optional asset metadata (`AssetPath`, `AssetId`, `Field`) when available.
- The run ends with a compact summary line in the format: `[Validation] N issue(s) / M error(s).`
- Editor and CI entry points continue using `report.HasErrors` / `report.ErrorCount` for pass/fail behavior.
